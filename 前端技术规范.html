1 前言

本文档的目的在于统一前端框架，其中某些点需要结合项目具体情况而定，所以只是建议性的，有任何意见大家一起讨论。

2 技术选型

2.1 框架套装

Angular + bootstrap：angular：1.6.2bootstrap：v3.3.0

优势：数据双向绑定，自定义指令+第三方组件，提高重用性

不足：应用层来说，不好处理dom非常复杂的场景，SEO不友好，游览器兼容性(IE 9+)

2.1 推荐组件

    1、angular-ui-router：angular官方推荐的第三方路由处理组件，ui-router是基于state的可嵌套的路由管理，可结合oclazyload实现懒加载以及预加载

版本号: "0.4.2"
官方地址：angular-ui-router

    2、angular-resource：基于Restful风格的$http服务，DMP项目对其简单地做了二次封装，针对CRUD提供不同的方法，并且自动设置token认证信息

版本号: "1.6.2"
官方地址：angular-resource

    3、ng-table：表格控件，功能比较齐全(排序，搜索，分页，嵌套，行内编辑，导出)，美观度也不错

版本号: "0.8.3"
官方地址：ng-table(ps:现在官方已经升级成了4.4.0版本)

    4、ng-echarts+echarts(3.0)：引入了百度echarts3.0，做图表功能，ng-echarts是针对原生echars的指令封装，操作更简单，不同图标只需要复制官方Option进行修改即可

ng-echarts版本号: "3.0.1"
echarts版本号: "3.0"
官方地址：ng-echarts
官方地址：echarts 3.0

    5、ng-dialog：弹窗控件，ngDialog体积小，功能强大，可以针对template设置conroller，处理非常灵活,支持自定义样式

版本号: "1.0.1"
官方地址：ng-dialog

    5、angular-ui-bootstrap：一款结合bootstrap以angular指令实现的UI标签组件，手风琴，轮播，日历，模态框，分页，标签页，进度条....

版本号: "2.5.0"
官方地址：angular-ui-bootstrap

    6、oclazyload：懒加载组件，可根据项目实际需求选择，spa项目在index页面加载全部文件也是可以的

版本号: "1.1.0"
官方地址：oclazyload

2.3 自动化构建工具

建议：npm+gulp (cnpm 可选)。

解释：bower 官方已经停止维护(站点还在国外，天朝你懂的),很多github项目也已经只提供npm的安装方式，所以慢慢弃用。gulp负责：

npm：包管理，npm3的扁平化依赖管理导致node_modules里有很多乱七八糟的文件，关于这一点，收敛一下自己的强迫症吧。
gulp：①：scss自动编译
  ②：游览器自动更新
  ③：上线版本号发布(解决在生产环境下更新后，客户端存在缓存的问题)
  ④：代码混淆

注意事项：

1、区分dependencies和devDependencies，前者是生产环境依赖项，后者是开发环境依赖项， 所以像gulp等构建工具放在后者，而项目使用的插件放在前者，判断标准：生产环境需不需要这个？
    ①、安装dependencies： 
npm install angular --save
    ②、安装devDependencies： 
npm install angular --save-dev
    ③、关于语义化版本号：假定当前版本是1.0.4(指的是你第一次用的时候，默认安装的就是latest version) 
只接受补丁包：1.0 或者 1.0.x 或者 ~1.0.4
只接受小版本和补丁包：1 或者 1.x 或者 ^1.0.4
接受所有更新：* or x
但是在实际应用中，我们并不需要它升级(甚至说任何改变)，所以请使用固定版本号，有两种办法：
方法一：npm install 后，手动修改package.json
方法二：npm install [packageName] --save[-dev]

总结：一定要--save,还有使用固定版本号

3 css规范

3.1 选择器命名

建议： CSS 选择器名要语义化。

解释：好的语义化选择器名，方便查找。选择器名尽量用单词、缩写语义化命名，多个单词用"-"中划线拼接。

/* 商品 */
.goods {
}
/* 商品列表 */
.goods-list {
}

3.3 空格

强制：选择器 与 { 之间必须包含空格。

示例：

.box {
}

强制： 属性名 与之后的 : 之间不允许包含空格， : 与 属性值 之间必须包含空格。

示例：

padding: 0;

3.4 选择器

强制：当一个 rule 包含多个 selector 时，每个选择器声明必须独占一行。

示例：

.post,
.page,
.comment {
    line-height: 1.5;
}

强制：当一个 >、 +、~ 选择器的两边各保留一个空格。

示例：

/* 推荐的 */
main > nav {
    padding: 10px;
}
label + input {
    margin-left: 5px;
}
input:checked ~ button {
    background-color: #69C;
}
/* 不推荐的 */
main > nav {
    padding: 10px;
}
label + input {
    margin-left: 5px;
}
input:checked ~button {
    background-color: #69C;
}

强制：属性选择器中的值必须用双引号包围。

解释：不允许使用单引号，不允许不使用引号。

示例：

/* 推荐的 */
article[character="juliet"] {
    voice-family: "Vivien Leigh", victoria, female
}
/* 不推荐的 */
article[character='juliet'] {
    voice-family: "Vivien Leigh", victoria, female
}

3.5 属性

强制：属性定义必须另起一行。

示例：

.box {
    padding: 0;
    margin: 0;
}

3.6 BEM命名示例

解释：BEM 是一种前端项目开发的方法学，由 Yandex 公司提出。BEM 的名称来源于该方法学的三个组成部分的英文首字母，分别是块（Block）、元素（Element）和修饰符（Modifier）。这三个不同的组成部分称为 BEM 实体。比较下面这两种不同的方式，BEM 的优点在于所产生的 CSS 类名都只使用一个类别选择器，可以避免传统做法中由于多个类别选择器嵌套带来的复杂的属性级联问题。在 BEM 命名规则中，所有的 CSS 样式规则都只用一个类别选择器。因此所有样式规则的特异性（specificity）都是相同的，也就不存在复杂的优先级问题。这可以简化属性值的层叠规则。代码清单中的命名规则的好处在于每个 CSS 类名都很简单明了，而且类名的层次关系可以与 DOM 节点的树型结构相对应。BEM 命名规则的可能的缺点在于 CSS 类名会比较长而且复杂。乍看之下，根据 BEM 命名规则产生的 CSS 类名都会很复杂，但实际上在熟悉了命名规则之后，可以很容易理解其含义。

示例：

/* 使用 BEM 命名规则的 CSS 类的 HTML 代码 */
.menu {
    list-style: none;
}
.menu-item {
    font-weight: bold;
}
.menu-item-selected {
    color: red;
}

<ul class="menu">
    <li class="menu-item menu-item-selected">Menu Item 1</li>
    <li class="menu-item">Menu Item 2</li>
    <li class="menu-item">Menu Item 3</li>
</ul>

/* 使用传统命名规则的 CSS 类的 HTML 代码 */
.menu {
    list-style: none;
}		 
.menu .item {
    font-weight: bold;
}		 
.menu .item.selected {
    color: red;
}		

<ul class="menu">
    <li class="item selected">Menu Item 1</li>
    <li class="item">Menu Item 2</li>
    <li class="item">Menu Item 3</li>
</ul>
4 JS代码规范

4.1 严格模式

强制： 严格模式可以规范大家的代码书写习惯，提高游览器编译效率

声明方式：下面是两种定义方式，前者是全文模式，后者是方法内严格模式，建议使用全文模式

xxx.js:
------------------
  'use strict'
  //coding here
------------------
aaa.js
------------------
  function(){
  	'use strict'
  	//code your method here
  };

4.2 controller定义

强制：必须使用数组式声明;控制名采用：模块名+'Ctrl'命名;多余的依赖请删除

解释：项目后期，代码需要做混淆处理，而内联式声明的控制器，混淆后，会发生编译错误;

//不推荐的
App.controller('forgot', function ($scope) {	
	
});
//推荐,命名语义化，并且采用数组式声明
App.controller('forgotCtrl', ['$scope',function ($scope) {
	
}]);

4.3 属性+方法定义

建议：

    ①：html上的变量定义在scope的属性上,而不是定义在scope上

    ②：scope的属性必须初始化,采用单复数的形式

    ③：function只要不是绑定在html的，不建议绑定在scope上

解释：①：前者可以避免域覆盖，规范变量使用

App.controller('forgotCtrl', function ($scope) {
	var temp = "面对疾风吧"; //最上面写全局变量
	
	$scope.userInfo = { 	//接着初始化scope的属性
		"name":"",
		"age":""
	}
	$scope.students Vs $scope.studentList  // 推荐前者，命名中，数据类型价值不高
	
	$scope.sayHello = function(){  //再接着定义scope的方法
	}; 	//请加上冒号，照顾一下使用WS的同事，-，-
	
	function whoAreYou(){ 	//最后写辅助方法
	};
});
5 项目结构

5.1 文件结构

建议： 采用模块化划分思想：业务+资源(图片，图标，字体...)+功能

解释：一个好的文件结构应该能够适应不同的项目规模，下面以DMP项目举例：

DMP:
---
   |node_modules
     	|..modules //插件
   |app
      |..config	//app配置，路由等
      |..css		//scss编译后的css
      |..pages
    	|..模块A
    	   |..**Ctrl.js
     	   |..**.html
      |..scss
      |..template
    |assets
     	|..images
     	|..fonts
   gulpfile.js
   index.html
   package.json
